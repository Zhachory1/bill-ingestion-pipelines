# Pre-Implementation Council Review

## Security Reviewer Perspective

### Issue #11 (Rate Limiting) - CRITICAL
**Concerns:**
1. ✅ Rate limiting is essential for public deployment
2. ⚠️ Per-IP limiting can be bypassed via proxies - consider adding session-based or API key-based limits as well
3. ✅ Input length bounds prevent DoS via large payloads
4. ⚠️ Need to ensure rate limit state persists (or document it's in-memory only)
5. ✅ HTTP 429 with Retry-After header is the right response

**Recommendations:**
- Use slowapi (FastAPI-compatible rate limiter)
- Default to conservative limits (e.g., 10 requests/minute per IP for chat)
- Add X-RateLimit headers for client awareness
- Document that in-memory limits reset on restart
- Consider Redis backend for production if needed later

### Issue #10 (LLM Validation)
**Concerns:**
1. ✅ Fail-fast validation prevents API key leakage to wrong provider
2. ✅ Provider enum prevents typos/injection

**Recommendations:**
- Validate at settings load time, not request time
- Log provider configuration on startup for debugging

### Issue #9 (Checkpoint)
**Security impact:** LOW - mostly reliability
**Recommendation:** Ensure checkpoint table has proper access controls

## Maintainer Perspective

### Architecture Decisions
**Issue #10:**
- ✅ Using Pydantic validators in settings is idiomatic
- ✅ Enum approach is clean and type-safe
- ⚠️ Consider adding model validation per provider (not just defaults)
- Document breaking change: now requires explicit LLM_PROVIDER

**Issue #9:**
- ✅ Using existing IngestCheckpoint model is right approach
- ✅ Storing commit SHA is more reliable than reflog
- ⚠️ Need migration path: what if someone has existing reflog-based workflow?
- Consider: graceful fallback if checkpoint missing (use HEAD~1 or full ingest)

**Issue #11:**
- ✅ Rate limiting should be opt-out, not opt-in (secure by default)
- ⚠️ Default limits should be documented clearly
- ✅ Configuration via env vars maintains 12-factor pattern

### Backward Compatibility
- Issue #10: BREAKING - requires explicit LLM_PROVIDER validation
  - Mitigation: Add clear error message, update .env.example
- Issue #9: NON-BREAKING - transparent to users
- Issue #11: BREAKING for automated scripts hitting API
  - Mitigation: Document how to disable for local dev
